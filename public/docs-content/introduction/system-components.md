# System Components

HoneyOps consists of two main components that work together:

1. **One or more Honeypot Endpoints**, deployed as either:
   - OVA virtual appliances, or
   - Windows agent deployments
2. **The HoneyOps Cloud Console**

These components communicate securely and exchange only minimal, security-focused telemetry.

---

## 1. Honeypot Endpoint (OVA or Windows)

HoneyOps can run in two deployment formats. Both represent the same managed honeypot object in the console.

### 1.1 OVA appliance

The OVA endpoint is a self-contained virtual machine delivered as an **Open Virtual Appliance (OVA)**.  
It is deployed inside your own infrastructure - datacenter, branch office, lab network or cloud VM hosting.

Key characteristics:

- Pre-hardened, low-maintenance image
- Simulated decoy services to attract attacker interactions
- Small runtime footprint
- Outbound-only communication to the console
- No access to unrelated customer production traffic

### 1.2 Windows agent

The Windows endpoint runs as a native Windows service and is remotely managed by the HoneyOps console.

Key characteristics:

- Native deployment on existing Windows hosts
- Same enrollment, policy, and telemetry lifecycle as OVA endpoints
- Centralized remote management from the HoneyOps console
- Suitable when a dedicated virtual appliance is not the preferred deployment model

### What the endpoint sends

- Heartbeat events: status, version, uptime, network reachability
- Incident telemetry: information about attacker probes and behavior
- Configuration acknowledgments and update status

### What the endpoint does *not* send

- No production traffic
- No credentials
- No file system contents from unrelated systems
- No packet captures unrelated to honeypot activity

This strict boundary ensures strong privacy and predictable behavior.

---

## 2. HoneyOps Cloud Console

The console is the central management plane. It runs in the HoneyOps cloud and provides:

### Management features

- Honeypot overview: health status, last heartbeat, version, metadata
- Activity: a consolidated record of actions, events and relevant honeypot activity
- Alert view: incidents generated by honeypot interactions
- Policies: configuration of detection rules and behavioral settings
- User and roles management: invite users, assign roles (owner, admin, operator, reader)
- Account and billing: manage organization profile and license usage

### Security properties

- Multi-tenant isolation between organizations
- API authentication using signed tokens per endpoint
- HTTPS-only communication with modern ciphers
- Role-based access control for all console actions

---

## 3. Communication Model

HoneyOps uses a **pull-based**, outbound-only model:

- The endpoint periodically contacts the console endpoint
- Telemetry and heartbeats are sent over secure HTTPS
- The console may return policy changes, license updates, or agent update metadata
- No inbound ports need to be opened on your network

This architecture avoids firewall complexity and works on almost any network, including restricted corporate environments.
